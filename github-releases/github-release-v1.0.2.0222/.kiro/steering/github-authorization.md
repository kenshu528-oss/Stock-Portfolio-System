# GitHub 授權規則 (GitHub Authorization Rules)

## 🚨 絕對禁止未經授權的 GitHub 操作

### 嚴格禁止的操作
- **絕對禁止** 在沒有用戶明確同意下上傳到 GitHub
- **絕對禁止** 在沒有用戶明確同意下修改 GitHub 倉庫
- **絕對禁止** 執行任何 `git push` 操作
- **絕對禁止** 建立或推送 Git 標籤
- **絕對禁止** 建立 GitHub Release
- **絕對禁止** 修改 GitHub Pages 部署

## ✅ 需要明確授權的操作

### 用戶必須明確說出以下詞語才能執行：
- **"可以上傳到 GitHub"**
- **"同意推送到 GitHub"** 
- **"可以發布"**
- **"同意發布到 GitHub"**
- **"可以 git push"**
- **"同意建立 Release"**

### 不算明確授權的表達：
- ❌ "好的"
- ❌ "OK"
- ❌ "沒問題"
- ❌ "可以"
- ❌ "繼續"
- ❌ 沉默或沒有回應

## 🔒 本地開發 vs GitHub 發布

### 允許的本地操作（無需授權）
- ✅ 修改本地檔案
- ✅ 本地 Git commit
- ✅ 建立本地 Git 標籤
- ✅ 版本號更新
- ✅ 功能開發和測試
- ✅ 本地備份

### 需要授權的 GitHub 操作
- 🔒 `git push origin main`
- 🔒 `git push --tags`
- 🔒 建立 GitHub Release
- 🔒 更新 GitHub Pages
- 🔒 修改 GitHub 倉庫設定
- 🔒 合併 Pull Request

## ⚠️ 違規後果

### 如果未經授權執行 GitHub 操作
1. **立即停止所有操作**
2. **向用戶道歉並說明情況**
3. **等待用戶明確指示**
4. **如有必要，協助回滾操作**

## 📋 操作前檢查清單

### 每次執行 Git 操作前必須確認：
- [ ] 這是本地操作還是 GitHub 操作？
- [ ] 如果是 GitHub 操作，用戶是否明確授權？
- [ ] 用戶是否使用了明確的授權詞語？
- [ ] 是否有任何疑慮需要再次確認？

## 🎯 正確的工作流程

### 開發階段（無需授權）
1. 修改程式碼
2. 本地測試
3. 本地 commit
4. 版本號更新
5. 功能驗證

### 發布階段（需要明確授權）
1. **向用戶說明要發布的內容**
2. **等待用戶明確授權**
3. **確認授權詞語正確**
4. **執行 GitHub 操作**
5. **確認發布成功**

## 💡 建議的溝通方式

### 請求授權時應該說：
```
"功能已完成並測試通過，是否可以上傳到 GitHub？"
"修復已完成，您是否同意推送到 GitHub 倉庫？"
"準備發布 v1.0.1.0002 版本，是否可以發布到 GitHub？"
```

### 不應該假設用戶同意：
```
❌ "我現在上傳到 GitHub"
❌ "準備推送..."
❌ "開始發布流程"
```

## 🔐 安全原則

### 寧可多問，不可擅自行動
- **寧可** 多次確認用戶意願
- **寧可** 等待明確授權
- **寧可** 保守行事
- **絕不** 擅自決定
- **絕不** 假設用戶同意
- **絕不** 忽略授權程序

---

**記住：GitHub 是用戶的資產，未經明確授權絕不操作！**
